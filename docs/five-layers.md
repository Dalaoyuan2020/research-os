# Research OS · 五层框架

> 在"用好别人的工具"之上,我把自己的科研方法蒸馏成五层,把零散工具焊成一条流水线。
> 这一页讲每层**做什么、用哪类 Skill、怎么和[「知识飞轮」](./philosophy.md)接上**。

## 总览

```
        读        沉淀        写         验证        管理
       L1  ──→   L2   ──→   L3   ──→   L4    ──→    L5
    文献阅读   知识库    文献写作   自动科研    项目管理
       └──────────── 闭环:读→沉淀→写→验证→管理 ───────────┘
```

| 层 | 角色 | 一句话 |
|---|---|---|
| **L1 文献阅读** | 读 | 把文献变成可检索、可速读的结构化输入 |
| **L2 知识库** | 沉淀 | 把读过的东西沉淀成可复用的知识网络 |
| **L3 文献写作** | 写 | 从改稿到投稿的全流程工程化 |
| **L4 自动科研** | 验证 | 用实验驾驭(Harness)把想法跑成结果 |
| **L5 项目管理** | 管理 | 多项目进度总控与任务编排 |

---

## L1 · 文献阅读层(读)

**做什么**:文献检索、结构解析、速读卡(时间 / 摘要 / 结论 / 主要方法 / 是否公开数据集)。把一篇 PDF 变成机器和人都能快速消化的结构化输入。

**用哪类 Skill**:文献检索、文档结构识别(标题/表格/图)、自动速读卡生成。

**吸收角度**:好的文献工具,值得学的是它怎么"压缩信息而不丢关键"。

## L2 · 知识库层(沉淀)

**做什么**:跨文献沉淀知识网络 —— 方法卡、结论卡、gap 卡互相双链。用的是 **Karpathy LLM Wiki 式**的轻量 markdown + 双向链接,不上重系统。

**用哪类 Skill**:知识卡生成、双链维护、文献情报迭代(检索→评分→入库→口味迭代)。

**和 L1 的区别**:L1 存"文献原件 + 整理卡",L2 存"跨文献的知识条目"。L2 的内容由 L3 的写作需求驱动产生 —— 有消费者才建,不空中楼阁。

## L3 · 文献写作层(写)

**做什么**:从改稿到投稿的全流程工程化 —— 改稿版本管理、模拟期刊评审、引用规范、去 AI 味、配图。

**用哪类 Skill**:改稿版本控制(Issue→PR→tag)、期刊评委(蒸馏目标期刊惯例当尺子)、引用格式转换、去 AI 味润色、科研配图。

**吸收角度**:这一层是把"写作经验"显式化成可调用的 Skill —— 正是「知识飞轮」最直接的体现。

## L4 · 自动科研层(验证)

**做什么**:想法(科学问题)→ 从 L1/L2 找办法 → 用**实验驾驭(Harness)**跑实验 → 得到结果。这是科研系统与研究对象合流的地方。

**用哪类 Skill**:实验 Harness、自迭代循环、停滞检测与心跳守护。

**定位**:全自动是远期目标;当前是**半自动**(人 + Agent 协作)的可验证路径 —— 定位对了,才不堵路。

## L5 · 项目管理层(管理)

**做什么**:多项目看板、进度总控、任务编排。让前四层在多个真实项目上并行不乱。

**用哪类 Skill**:多项目进度面板、任务编排、交接卡(跨会话/跨 Agent 的状态恢复)。

---

## 为什么是"操作系统"

这五层不是五个孤立工具,是一条闭环:**读 → 沉淀 → 写 → 验证 → 管理**。

更关键的是:**系统本身也在吸收[热榜](../README.md#-基础--学术-ai-工具热榜)里别人的好工程,自我迭代。** 热榜发现一个更好的引用工具,就接进 L3;发现一个更稳的实验框架,就接进 L4。

这就是从"论文写作工具集"升格为"科研操作系统"的原因 —— 它不是写死的,它在长。
